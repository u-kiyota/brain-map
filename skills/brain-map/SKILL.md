# brain-map

個人の思考プロセス・作業プロセス・スキル・口癖を体系化し、AI が「その人を再現する」ための知識ベースを構築するスキル。

## このスキルが解決すること

組織で特定のメンバーの判断・指摘・コミュニケーションパターンを AI に学ばせたいケースは多い。だが Slack や Notion に散らばった発言を都度収集するのは現実的でない。このスキルは情報源を一度登録し、差分更新で蓄積していくことで、AI が参照しやすい単位（カテゴリ別マークダウン）で知識を構築する。

「AI への入力」が前提なので、各ファイルは肥大化させず、独立して読める単位に分割するのが原則。さらに、観測した発言をすべて等価に積み上げるのではなく、**重要度（サリエンス）で取捨選択・格付け**して「本人が本当に大切にしていること」が前面に出るようにする（後述「重要度（サリエンス）モデル」）。

## 実行モデル（サブエージェント前提）

情報源の生データ（Slack 全発言・Notion DB・Obsidian ノート群など）をメインのコンテキストに全部読み込むと破綻する。そのため、**重い取得・抽出はサブエージェントの内側に閉じ込め、メインには圧縮済みダイジェストだけ返す** map-reduce 構造で実行する。

- **基本方針**: 常に「1 情報源 = 1 サブエージェント」。小規模ソースでも同じ。
- **不変条件 1（匿名化の中央集権）**: サブは匿名 ID を採番しない。生テキストを読んで観察を抽出し、本文中の人物・組織固有名詞は実名のまま引用に残しつつ「登場人物リスト」「組織・固有名詞リスト」を返す。ID 採番・汎化・最終置換・`_anonymization.yaml` 更新はメインが独占する。これがないと同一の人物・会社に別 ID が振られ一貫性が壊れる。
- **不変条件 2（書き込みの中央集権）**: サブは出力ファイル（リポジトリ直下のブレインマップ md・`index.md`・`_manifest.yaml` など）を一切書かない（読取・抽出専用）。カテゴリファイルの作成・統合・`index.md`・`_manifest.yaml` 更新は全てメインが行う。並列書込の競合を防ぐため。

ダイジェストの schema、サブエージェントへのプロンプト雛形、合成（Reduce）手順、検証サブエージェントは `references/subagents.md` を参照。データ取得を伴う全モード（初回作成・更新・再整理の一部）はこの実行モデルに従う。

## 重要度（サリエンス）モデル

**最重要の設計方針。** 発言を片端から取り込むとファイルが肥大化し、「一度しか言っていない些末な発言」と「何度も繰り返し強く主張している核心」が同じ重みで並んでしまう。これでは AI がその人を再現するとき、何を優先すべきか分からない。そこで観察ごとに**重要度（サリエンス）を判定し、格付け・並べ替え・退避**を行う。

### 重要度を決める 3 シグナル

観察（principle 単位）ごとに、次の 3 つから重要度を推定する。

- **頻度（recurrence）**: 同じ趣旨が複数のメッセージ・複数の情報源・複数の時期にわたって繰り返し現れるほど重要。一度きりは弱い。
- **強度（intensity）**: 言い方の強さ。「絶対」「必ず」「これだけは」等の強調語、同一文内での反復、感情を伴う表明、明示的な原則宣言は強い。
- **一貫性（consistency）**: 文脈や時期をまたいでブレずに保たれているか。状況限定の一回限りの反応は弱い。行動が繰り返し伴っているものは強い。

これらは内部の判断材料であり、**出力本文にスコアや回数は書かない**（重みは節構造と並び順だけで表現する）。判断の根拠となる頻度・情報源・最終観測などのメタは `_salience.yaml` に保持する（出力ファイルには出さない管理ファイル）。

### 3 階層

| 階層 | 定義 | 出力先・扱い |
|------|------|------|
| **核（コア）** | 高頻度＋強い強度＋一貫。その人を最も特徴づける。 | 各カテゴリファイル冒頭の「コア」節。`index.md` に凝縮。並び順は重要度降順。 |
| **補助（サポート）** | 複数回観測されるが中心的ではない。状況依存だが意味はある。 | 各カテゴリファイルの「補助的なパターン」節。コアの下。 |
| **周辺（ペリフェラル）** | 一度きり・弱い強度・状況限定で、本人像の理解にほぼ寄与しない些末な観察。 | **本体ファイルには置かず `_archive.md` に退避**。 |

### アーカイブ（退避）の原則 — 「削除ではなく退避」

低重要度の些末な観察は本体カテゴリファイルから除去するが、**完全には消さず `_archive.md` に出自付きで退避する**。これにより、

- 本体（AI が読む知識ベース）はスリムに保たれ、核が際立つ。
- 情報は失われない（後から参照・復元できる）。`git` と合わせて常に戻せる。
- **再出現したら昇格する**。アーカイブは墓場ではなく保留領域。更新時に同趣旨が再び観測され頻度がしきい値を超えたら、`_archive.md` から本体の補助／コアへ引き上げる。

退避の判断目安（すべて満たすとき周辺＝退避候補）:
1. 観測が 1 回きり（単一メッセージ・単一情報源）
2. 強調語・反復・感情を伴わない
3. 特定の状況に限定され、本人の思考・価値観・スタイルの一般化に寄与しない

迷ったら退避しない（補助に残す）。退避は本人像の解像度を上げるための整理であり、情報の切り捨てではない。

### 各モードでの適用

- **初回・更新**: Reduce 時にメインが各観察の重要度を判定し、`_salience.yaml` を更新。核／補助は本体へ、周辺は `_archive.md` へ振り分ける。更新で既存観察と一致したら頻度を加算し、しきい値超えで昇格。詳細は `references/subagents.md`。
- **再整理**: 取得を伴わず、既存ファイルを重要度で再格付けする。肥大したファイルから周辺観察を `_archive.md` へ退避し、核を冒頭へ並べ替える。詳細は `references/reorganize.md`。

## 保存先

このスキルは持続的フォルダを前提とする（manifest を読み込んで差分更新するため）。**1 リポジトリ＝1 ブレインマップ**を前提とし、ブレインマップファイル（`index.md`・カテゴリ別 md・`_manifest.yaml`・`_anonymization.yaml`）はすべて **リポジトリ直下（ルート）** に置く。GitHub Pages をルートフォルダから配信できるようにするため、slug サブフォルダは作らない。

### 検出ロジック

開始時に以下の順で保存先を決定する：

1. **マウント済みフォルダ直下に `_manifest.yaml` が存在する** → そのフォルダを repo ルートとみなし、ルート直下を保存先として使う（更新・再整理モード）
2. **マウント済みフォルダ直下に `index.md` やカテゴリ別 md・`.git` が存在する** → 既存のブレインマップ repo とみなし、ルート直下を使う
3. **マウント済みフォルダがあるが空（初回作成）** → ユーザーに「このフォルダ直下にブレインマップファイルを作成してよいか」を確認し、ルート直下を使う
4. **未マウント** → `mcp__cowork__request_cowork_directory` でフォルダ依頼。ユーザーが拒否したら `outputs/` 直下に保存し、「このセッション限定の出力になる」旨を明示

1 リポジトリには 1 人分のブレインマップのみを置く。別の対象者を扱う場合は別リポジトリ（別フォルダ）を用意する。

### 推奨セットアップ

このスキルは git 管理リポジトリ（例: `yukiyota-neural-circuit`）と組み合わせて使うことを想定。リポジトリの構造例：

```
<repo>/                          # = 1 ブレインマップ
├── README.md
├── .gitignore
├── index.md                     # 全体サマリ・接する際の注意点（核を凝縮）
├── summary.html                 # index.md の人間閲覧用 HTML（更新時に必ず再生成）
├── _manifest.yaml               # 情報源と最終取得状態
├── _anonymization.yaml          # 匿名化マッピング（gitignore推奨）
├── _salience.yaml               # 重要度メタ（頻度・情報源・最終観測。出力には出さない）
├── _archive.md                  # 退避した低重要度の観察（復元・昇格用の保留領域）
├── work-philosophy.md           # カテゴリ別ファイル（AI が読む本体）
├── work-philosophy.html         # 各カテゴリの人間閲覧用 HTML（render_html.py が自動生成）
├── communication-style.md
├── communication-style.html
├── ...
└── skills/brain-map/            # このスキルの本体（scripts/render_html.py を同梱）
```

`_anonymization.yaml` は実名を含むため、共有 repo では `.gitignore` 対象にする。`_salience.yaml`・`_archive.md` は管理用で AI の知識ベース本体ではない（`_archive.md` も匿名化済みにする）。`summary.html` と各 `<category>.html` は人間閲覧用の派生物で、`scripts/render_html.py` から再生成できる。

## 人間用 HTML の自動出力（summary.html ＋ カテゴリ別 HTML）

`index.md` やカテゴリ別 md は AI が読む前提のマークダウンで、人が見るには読みにくい。そこで **ブレインマップ本体を作成・変更したときは必ず、人が見やすい HTML 一式を出力する**。HTML は次の 2 種類:

- **`summary.html`** — `index.md` の人間閲覧用ランディングページ（サマリ・役割・強み・思考の特徴・コミュニケーションの特徴・接する際の注意点・知識ベースへのリンク）。テンプレートに index.md の実内容を流し込んで手動でキュレーションする。
- **`<category>.html`** — 各カテゴリ別 md（`work-philosophy.md` など）を 1 対 1 で HTML 化したページ。`scripts/render_html.py` で自動生成する。上部に「← サマリへ戻る」ナビと他カテゴリへのピルリンク（現在ページをハイライト）を持ち、summary.html を起点にサイトとして回遊できる。

不変ルール（初回作成・更新・再整理のいずれのモードでも、本体 md を書いたら必ず実行する）:

1. **カテゴリ HTML を生成する** - `scripts/render_html.py`（このスキル同梱）をリポジトリ直下で実行する:

   ```bash
   python3 skills/brain-map/scripts/render_html.py <repo_root>
   ```

   `_` 始まり・`README.md`・`index.md` を除く全カテゴリ md を `<同名>.html` に変換する（`markdown` パッケージが無ければ自動導入を試みる）。各ページは summary.html と統一した配色・カード調の共通 CSS でラップされ、summary.html へ戻るナビと兄弟カテゴリへのリンクが付く。
2. **summary.html を生成する** - 確定後の `index.md` の内容をもとに、リポジトリ直下に `summary.html` を出力する（外部依存なしの自己完結した単一 HTML。テンプレートは `references/templates.md`「summary.html テンプレート」）。**「知識ベース」セクションの各カードは `<category>.html`（.md ではなく）へリンクする**ので、閲覧者は HTML だけで回遊できる。閲覧者がぱっとスキャンできるカード／セクション構成にする。
3. **summary.html のリンクを .html に揃える** - `render_html.py` は仕上げに summary.html 内のカテゴリへの `href="./xxx.md"` を `href="./xxx.html"` に書き換える（手書きで .md のまま残っていても自動補正される）。`index.md`（AI 用）のカテゴリリンクは `.md` のまま維持する。
4. **index.md にリンクを記載する** - `index.md` 冒頭（タイトル直下）に、人間用サマリ HTML への誘導リンクを必ず入れる。形式は次の 1 行:

   ```markdown
   > 📄 **人間用サマリ（HTML）はこちら → [summary.html](./summary.html)**（ブラウザで開くと見やすく整理されています）
   ```

   既にこの行があれば重複させない（リンク先・文言が変わらなければそのまま）。
5. **整合性を保つ** - 本体 md を更新したら HTML 一式（`summary.html` ＋ 各 `<category>.html`）も作り直す（古いまま残さない）。HTML は md の派生物であり、両者は常に一致させる。匿名化のルールは md と同じく HTML 全部に適用する（`render_html.py` は匿名化済み md をそのまま変換するので新たな実名混入はないが、検証対象には含める）。
6. **ユーザーへの提示** - `present_files` で共有する際は `index.md`・`summary.html` と、変更があったカテゴリの `<category>.html` も提示する。

`summary.html`・`<category>.html` は AI の知識ベース本体ではなく人間閲覧用の派生物。`_manifest.yaml` の `files` には含めても含めなくてもよいが、リポジトリ直下に置き GitHub Pages から `summary.html` を入口に閲覧できるようにする。`render_html.py` の詳細は `references/templates.md`「カテゴリ HTML と render_html.py」を参照。

## ワークフロー判定

まずユーザーの依頼意図と、リポジトリ直下の `_manifest.yaml` の有無で 3 モードを判定する：

- manifest が存在しない → **初回作成モード**
- manifest が存在し、新規データ取得を伴う → **更新モード**
- manifest が存在し、取得を伴わず構造を整え直す → **再整理モード**

判定の優先順位：
- 「新しく作って」と言われたら強制的に初回モード。
- 「再整理して」「カテゴリを整理し直して」「体系を見直して」「ファイルを分割して」「index を作り直して」など、新情報の取得ではなく既存の整理を求める依頼 → **再整理モード**（`references/reorganize.md` 参照）。
- 「○○さんのブレインマップを更新して」や、対象者名がすでに manifest にあって新しい発言・記述を取り込みたい依頼 → 更新モード。
- 取得と再整理の両方を求められたら、更新モードを実行してから再整理モードを続けて実行する。

## 初回作成モード

データ取得は実行モデル（サブエージェント）に従う。詳細は `references/subagents.md`。

1. **対象者名の確認** - ユーザーに対象者名（表示用）を聞く。slug は `_manifest.yaml` 内の識別子としてのみ使う（未指定なら表示名から自動生成）。出力ファイルはルート直下に置くため、ディレクトリ名としての slug は不要。
2. **情報源の聞き取り** - 「どの情報源から抽出しますか？」と聞く。各情報源について必要な識別子を取得（後述）。
3. **Map: ソース別サブエージェント起動** - 情報源ごとに 1 サブエージェントを起動し、生データ取得＋観察抽出を委ねる。各サブは観察ダイジェスト＋登場人物リストを返す（実名のまま）。雛形は `references/subagents.md`。
4. **Reduce: 集約・匿名化** - メインが全ダイジェストを集約。新規の人物に匿名 ID、新規の組織・固有名詞に汎化ラベルを採番し `_anonymization.yaml`（`anonymized`／`entities`）を作成、ダイジェストへ最終置換を適用。
5. **重要度判定と振り分け** - 各観察を頻度・強度・一貫性から核／補助／周辺に格付けし `_salience.yaml` を作成（「重要度（サリエンス）モデル」参照）。
6. **カテゴリ分類と執筆** - 核／補助はカテゴリ別マークダウンへ（コア節→補助節、重要度降順）、周辺は `_archive.md` へ退避。書き込みはメインのみ。
7. **index.md 合成（サブエージェント）** - 確定したカテゴリファイル群を読ませ、index.md 草案を返させる。サマリ・特徴に加え「接する際の注意点」（大切にしていること／誤解されやすいこと／その他の協働上の注意点）を必ず含める。index には核を凝縮し、周辺・補助の些末は載せない。書き込みはメイン。詳細は `references/subagents.md`「index.md 合成サブエージェント」。
8. **HTML 一式の出力＋リンク追加** - `scripts/render_html.py` を実行して各カテゴリ md を `<category>.html` に変換し、確定後の index.md をもとに `summary.html` を生成する（知識ベースのカードは `<category>.html` へリンク）。index.md 冒頭に summary.html へのリンク行を記載する。詳細は「人間用 HTML の自動出力」と `references/templates.md`。
9. **検証** - 匿名化漏れを検証（検証サブエージェント推奨）。`_archive.md`・`summary.html`・各 `<category>.html` も対象に含める。
10. **manifest 保存** - 取得済み情報源と最終取得位置を記録。
11. **ユーザーへ提示** - `present_files` で index.md・summary.html と主要ファイル（生成した `<category>.html` を含む）を共有。

## 更新モード

差分取得も実行モデル（サブエージェント）に従う。詳細は `references/subagents.md`。

1. **manifest 読み込み** - `_manifest.yaml` から対象者名と各情報源の前回取得位置を、`_anonymization.yaml` から既知の匿名化マップを、`_salience.yaml` から既存観察の重要度メタを取得。
2. **追加情報源の確認** - ユーザーに「情報源を追加しますか？」と聞く。追加があれば manifest に登録。
3. **Map: 差分取得サブエージェント起動** - 情報源ごとに 1 サブエージェントを起動。前回カーソル以降のみ取得させ、既知の匿名化マップも渡す。差分がなければそのソースはスキップ。
4. **Reduce: 集約・匿名化** - メインが新規ダイジェストを集約。新規人物に ID 採番、新規の組織・固有名詞に汎化ラベルを採番、`_anonymization.yaml`（`anonymized`／`entities`）更新、最終置換を適用。
5. **重要度の更新と振り分け** - 新規観察が既存観察（本体・`_archive.md` 双方）と一致したら `_salience.yaml` の頻度を加算し、しきい値超えなら昇格（周辺→補助→コア）。新規一回きりの些末は周辺＝`_archive.md` へ。詳細は `references/subagents.md`。
6. **既存ファイルへのマージ** - 核／補助を既存カテゴリへ統合（重複回避・文脈差は残す。各ファイル内はコア節→補助節、重要度降順）。昇格した観察は `_archive.md` から本体へ移す。新カテゴリが必要ならメインが新ファイル作成。
7. **index.md 再合成（サブエージェント）** - マージ後の最新カテゴリファイル群を読ませ、index.md 全体を作り直した草案を返させる。末尾に追記するのではなく、更新後のファイルを根拠に「接する際の注意点」（大切にしていること／誤解されやすいこと／その他の協働上の注意点）まで含めて再合成する。index には核を凝縮する。書き込みはメイン。詳細は `references/subagents.md`「index.md 合成サブエージェント」。
8. **HTML 一式の再出力＋リンク確認** - 変更があった（および新設の）カテゴリについて `scripts/render_html.py` を再実行して `<category>.html` を作り直し、index.md をもとに `summary.html` も同じ内容に作り直す（古いまま残さない）。index.md 冒頭のリンク行を確認・維持する。詳細は「人間用 HTML の自動出力」。
9. **検証** - 匿名化漏れを検証（`_archive.md`・`summary.html`・各 `<category>.html` も対象）。
10. **manifest 更新** - 取得・処理が完了したソースのみ、最終取得位置とタイムスタンプを更新。
11. **ユーザーへ提示** - 変更があったファイル（index.md を変更したら summary.html と該当 `<category>.html` も）を `present_files` で共有。

## 再整理モード

新規データ取得を伴わず、リポジトリ直下の既存ブレインマップファイルだけを対象に構造を整え直すモード。蓄積でカテゴリ境界が崩れたとき（肥大ファイル・薄いファイル乱立・同趣旨の分散・index の密度低下）、および**重要度の格付けが崩れたとき（些末な観察が核に混じって肥大化）**に使う。詳細手順は `references/reorganize.md`。

1. **安価な俯瞰調査** - 全文は読まず、各カテゴリファイルの行数と見出しだけ抽出。`index.md` のみ全文読む。
2. **新分類の決定** - 分割・統合・再配置・重複統合の移送マップを設計。破壊的変更ならユーザーに提示して合意を取る。
3. **移送・執筆と重要度の再格付け** - 目標カテゴリごとに 1 サブエージェントを起動し、移送元ブロックを読んで、コア／補助に格付けした整形本文＋退避候補（周辺）を返させる。メインが本体へ書き込み、周辺は `_archive.md` へ退避、`_salience.yaml` を更新。空になった元ファイルは削除し `_manifest.yaml` の `files` を更新（対応する `<category>.html` も古いものを削除する）。
4. **index.md 再生成（サブエージェント）** - 新構成のカテゴリファイル群を読ませ、サマリ・リンク一覧・「接する際の注意点」（大切にしていること／誤解されやすいこと／その他の協働上の注意点）まで作り直した草案を返させる。書き込みはメイン。詳細は `references/subagents.md`「index.md 合成サブエージェント」。
5. **HTML 一式の再出力＋リンク確認** - `scripts/render_html.py` を再実行して新構成の全 `<category>.html` を生成し、index.md をもとに `summary.html` も同じ内容に再生成する。index.md 冒頭のリンク行を維持する。詳細は「人間用 HTML の自動出力」。
6. **匿名化の再検証** - 移送・退避で実名が紛れ込んでいないか検証サブエージェントで走査（`_archive.md`・`summary.html`・各 `<category>.html` も対象）。
7. **差分提示** - 増減・新設・削除・退避を要約し、主要ファイル（index.md・summary.html・該当 `<category>.html` 含む）を `present_files` で共有。取得カーソルは触らず `updated_at` のみ更新。

再整理は移送・統合・分割・**重要度に基づく退避**であって完全削除ではない。低重要度の観察は `_archive.md` に退避され情報は失われない（後から昇格・復元できる）。実行前に git がクリーンであることを確認し、いつでも戻せる状態を保つ。

## 情報源別の取得ガイド

ユーザーの環境では特に Slack / Notion / Obsidian / Gmail / Google Drive / GitHub（Slack 経由）が候補。

### Slack
- 識別子: チャンネル ID もしくは名前、ユーザー ID、対象期間
- 取得: `slack_search_public_and_private` で `from:<user_id>` を指定するのが基本。チャンネル全件なら `slack_read_channel`。スレッド深掘りは `slack_read_thread`
- DM: ユーザー ID を `from:` 指定で検索
- GitHub bot 連携メッセージ（PR レビュー等）も対象。`from:` で対象者を絞り、コードレビューコメントを抽出
- 差分キー: メッセージの `ts`（Slack タイムスタンプ）
- manifest に保存: `last_message_ts`
- 注意: 取得した本文に他メンバーへのメンション（`<@U…>`）があれば、ユーザー解決のうえ匿名化

### Notion
- 識別子: ページ URL、データソース URL、対象期間
- 取得: `notion-fetch` で URL を指定。データベースなら data_source を fetch し、各ページの ID を fetch
- 差分キー: 各ページの `last_edited_time`
- manifest に保存: ページごとに `last_edited_time`
- 振り返りシートなど DB の場合は、ページ単位で個別に last_edited_time を管理

### Obsidian
- 識別子: vault 内のパス（ファイルまたはディレクトリ）
- 取得: ディレクトリなら `obsidian_list_files_in_dir` → `obsidian_batch_get_file_contents`。ファイル単体は `obsidian_get_file_contents`
- 差分キー: ファイル本体のハッシュ（`hashlib.md5(content).hexdigest()`）。Obsidian の更新時刻 API が安定しない場合はハッシュで判定
- manifest に保存: パスごとに `content_hash`

### Gmail
- 識別子: 検索クエリ（`from:user@example.com` 等）、対象期間
- 取得: `search_threads` でクエリ実行
- 差分キー: メッセージの `internalDate` または ID
- manifest に保存: `last_message_id` + `last_internal_date`

### Gmail（Gemini 会議メモ → 会議文字起こし）
Google Meet の AI メモ機能が `gemini-notes@google.com` から送ってくる会議メモを起点に、**会議の文字起こしから対象者本人の発言だけ**を抽出する専用パターン。会議が音声込みで記録されるため、Slack/Notion では拾えない「口頭での思考・口癖・レビュー観点・FB の語り口」を取れるのが価値。

- **識別子**: 検索クエリ `from:gemini-notes@google.com`、対象期間。件名は「メモ: 「{会議名}」（YYYY年M月D日）」形式。
- **取得フロー**（1 会議 = 1 サブエージェント。詳細は `references/subagents.md`）:
  1. `search_threads(query="from:gemini-notes@google.com")` で会議を列挙。件名から会議種別・日付を判別し、**対象者の発言が濃い会議に絞る**（例: 定例・レビュー会・1on1・経営/部門会議。純粋な事務連絡・面接・入社手続き等は除外）。
  2. 各 thread を `get_thread(messageFormat="FULL_CONTENT")` で取得し、`htmlBody`/`plaintextBody` から会議メモ Doc のリンク `docs.google.com/document/d/<DOC_ID>` を抽出（「会議メモを開く」ボタンの URL）。
  3. Drive `read_file_content(fileId=<DOC_ID>)` で Doc 全文を取得する。Doc 後半の `# 文字起こし` セクションに `**発言者名:**` ラベル付きで全発言が並ぶ。
  4. **発言者が明確に対象者本人とラベルされた発言のみ**を抽出する（曖昧・混線箇所は使わない）。冒頭の要約部の記述は補助に留め、引用は実発言から取る。
- 差分キー: メッセージの `internalDate`
- manifest に保存: `last_internal_date`（Gemini 会議メモ用に、通常 Gmail とは別カーソルで管理）
- 注意: 文字起こしには他参加者の発言も含まれる。対象者本人の発言だけを抽出し、本文中の他者名はメインで匿名化する。

# 再整理モード

既存のブレインマップを、**新しい情報源取得を伴わずに**構造面から整え直すモード。蓄積が進むとカテゴリ境界が崩れる（1 ファイルが肥大、薄いファイルが乱立、同趣旨が複数ファイルに分散、index がサマリとしての密度を失う）。また、**些末な観察が核に混じって肥大化し、本人が大切にしていることが埋もれる**こともある。そのときに使う。

## トリガー

- 「清田のブレインマップを再整理して」「カテゴリを整理し直して」「体系を見直して」
- 「ファイルが大きくなってきたから分割して」「index を作り直して」
- 「些末な発言が増えて肥大化している、大切なものが分かるよう整理して」「重要度で格付けし直して」

情報取得を伴わないので、コネクタ接続は不要。対象はリポジトリ直下（ルート）のローカルファイルのみ。

## 原則

- **情報を失わない**。再整理は移送・統合・分割・**重要度に基づく退避**であって、完全削除ではない。重複の解消で消えるのは「同趣旨の重複表現」だけ。低重要度の周辺観察は本体から外すが `_archive.md` へ退避するので失われない（再出現で昇格可能）。
- **重要度で格付けし直す**。SKILL.md「重要度（サリエンス）モデル」に従い、各観察を核／補助／周辺に再判定。核を各ファイル冒頭へ、補助をその下へ、周辺を `_archive.md` へ。`_salience.yaml` があれば頻度を根拠に、無ければ本文の強調・反復・複数箇所での出現から推定する。
- **git で安全網を張る**。実行前にコミット済み（またはクリーンな作業ツリー）であることを確認。作業後に差分を提示し、ユーザーがいつでも戻せる状態にする。
- **匿名化・汎化を再保証する**。移送後も `_anonymization.yaml` の `anonymized`（人名）と `entities`（会社名・製品名・案件名・社内固有用語）に照らして漏れがないか検証する。
- **メインが書き込みを独占**。サブエージェントは読取・抽出・提案のみ（[[subagents]] の不変条件 2 と同じ）。

## 手順

### 1. 安価な俯瞰調査（メイン）

全文は読まない。コンテキストを節約しつつ全体像だけ掴む。

- `_manifest.yaml` の `files` 一覧を読む。
- 各カテゴリファイルの **行数** と **見出し（## / ###）だけ** を抽出する（例: bash で `for f in <repo>/*.md; do echo "== $f =="; wc -l "$f"; grep -nE '^#{1,3} ' "$f"; done`）。
- `index.md` は全文読んでよい（サマリなので軽い）。

この時点で「どこが肥大か」「どこが薄いか」「見出しの重複・分散」が見える。

### 2. 新分類の決定（メイン）

俯瞰結果から、目標のカテゴリ構成と移送マップを設計する。判断基準は SKILL.md と同じ「AI が特定の状況で参照したい単位か」。

典型的な操作:
- **分割**: 300 行超 / 2 つ以上の独立テーマが同居 → 複数ファイルへ。
- **統合**: 観測量が薄く単独で読む意味が乏しい → 近接カテゴリへ吸収。
- **再配置**: 見出し単位で本来別カテゴリに属する塊を移送。
- **重複統合**: 同趣旨の記述を 1 箇所に集約し、言い回しはバリエーションとして並列化。
- **重要度の再格付け**: 各ファイルを「コア」節→「補助的なパターン」節に再構成し、重要度降順に並べ替える。
- **退避**: 一回性・些末な周辺観察を `_archive.md` へ移し、本体をスリム化する（削除ではない）。

移送マップを明示的に作る（どの見出しブロックがどのファイルへ行くか）。これがサブエージェントへの指示書になる。

```yaml
# 移送マップの例
target_categories:
  - communication-style.md      # 既存維持
  - feedback-style.md           # 既存維持
  - project-management.md       # task-management.md を吸収
  - code-review.md              # 新規: work-approach.md 内のレビュー観点を集約
moves:
  - from: work-approach.md / "## コードレビューの観点"
    to:   code-review.md
  - from: task-management.md / "全体"
    to:   project-management.md / "## タスク管理"
deletes_after_merge:
  - task-management.md          # 中身を移送し終えたら空になるので削除
dedup:
  - "communication-style.md と feedback-style.md に重複する『結論先出し』を前者に集約"
archive:
  - "communication-style.md の一回性の絵文字癖を _archive.md へ退避（周辺）"
```

破壊的な構成変更になる場合は、移送マップをユーザーに提示して合意を取ってから実行する。

### 3. 移送・執筆（サブエージェント → メインが書込）

肥大ファイルの読み込みでメインのコンテキストを汚さないよう、**目標カテゴリごとに 1 サブエージェント**を起動し、「移送元の指定ブロックを読み、目標ファイル向けに整形した本文を返す」役を担わせる。サブはファイルを書かない。

サブへの指示雛形:

```

目標カテゴリ「{target}.md」に集約すべき内容を組み立ててください。

【読むファイルと該当ブロック】
{移送マップの from 一覧。ファイルパスと見出し}

【あなたの責務】
1. 指定ブロックを読み、目標カテゴリにふさわしい内容だけを抽出・整形する。
2. SKILL.md のカテゴリファイル書き方原則に従う（概要→コア節→補助的なパターン節→典型発言→アンチパターン、
   抽象原則＋具体例＋文脈付き引用、1 ファイル 300 行以内、階層 2 レベルまで、スコア・回数は書かない）。
3. 各項目を重要度で格付けする。高頻度・強調・複数箇所での反復・一貫＝コア、複数回だが中心的でない＝補助、
   一回性・弱い・状況限定で本人像にほぼ寄与しない＝周辺。コア→補助の順、各節内も重要度降順に並べる。
4. 重複する同趣旨の記述は 1 つに統合し、異なる言い回しはバリエーションとして並列に残す。
5. 周辺と判定した観察は本文に含めず、末尾の「退避候補」に出自（元ファイル・該当箇所）付きで列挙する。
6. 情報を失わない。判断に迷う内容は捨てず、コア/補助のいずれか最も近い節に含める（迷ったら退避しない）。

【禁止事項】
- ファイルの書き込み・編集はしない（メインが行う）。`_archive.md` も書かない（退避はメインが行う）。
- 匿名化マップを変更しない。新たな人名や会社特定可能な固有名詞（会社名・製品名・案件名・社内固有用語）が実名で見つかったら報告だけする。
- 観測のない内容を創作しない。

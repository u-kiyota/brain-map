コードレビュー・開発の進め方

清田が GitHub の PR レビューと Slack で見せる、ソースコードレビューの観点・指摘パターンと、開発の進め方（Git運用・テスト・本番運用・AI活用）。情報源は GitHub の PR レビュー履歴（omeroid org の各リポジトリ＋クライアントC の案件α周辺、2023–2026 の3年分。主に Go の社内システムA・C・D と、クライアントC の Nuxt/Vue + AWS Amplify GraphQL）と、Slack の各PJチャンネルの発言。重要度の高い「コア」を先頭に置く。

仕様書・要件定義・テスト仕様書・DB設計など上流／設計のレビューは別ファイル [[requirements-and-design]] に整理済み。本ファイルは実装コードのPRレビューと開発オペレーションに焦点を当てる。

コア — レビューの基本姿勢（最重要）

高頻度・強い確信で一貫して観測される、清田のレビューを最も特徴づける姿勢。重要度の高い順。

  • レビューの第一目的は「仕様を満たしているか」。ソースの良し悪しはその次。コードがどれだけ綺麗でも仕様を満たしていなければレビューを通す意味がない、という優先順位を明確に持つ。
  • GitHubの差分だけを見るのはレビューではない。diff 上で「変なことをしていない」の確認だけでは不十分。差分に現れない非実装箇所まで意識し、コード全体で横串の整合（既存ロジック・他箇所との一貫性）が取れているかを確認して初めてレビューと言える。

    「単体PR・ソースレベルでOKでなく、全体として仕様を満たせてるか、他に抜け漏れがないかをまずチェックしなきゃいけない」

  • 「目で追って確認した気になる」のを最大の敵とみなし、1行ずつ読ませる。

    「確認する時に、目で追って確認した気になっているのが原因だと思う」（被レビュー側の弁）に対し 「はい。1行1行丁寧に読んでチェックしてください。」

  • 一度受けた指摘は二度と出させない（再発防止）。同じ指摘の繰り返しを強い警告として扱う。同一PR内の再発は仕組み（エディタ設定）で潰せと指示する。

    「一度受けた指摘をきちんと覚えておくor漏れないようにメモるなどの対策をすることが大切です」 「末尾の改行だけ入ってないファイルある。このPR内で何度も指摘しているので、ちゃんと確認するか、エディタの設定で防いでください。」

  • 指摘には必ず返信させ、対応有無を可視化させる。レビュアーが修正を追えない状態を嫌う。

    「PRの私的には基本的にコメント返してもらえますか? どう対応したのかわからないので。簡易な指摘であれば修正しましたとか、doneとかだけでも大丈夫です。」

  • 「横串で修正してください」が口癖。指摘した1箇所だけでなく、同種のコード全てに直しを波及させる。
  • 境界・エッジケースを先回りして問う。実データの動きを想像して破綻条件を突く。

    「2mにしたら地面が2m以内だとずっと検知する？」「後から画像を変えたら保存済み座標どうなる？」

レビュー指摘の頻出観点（コード）

コアの姿勢が具体的なコード指摘として現れるパターン。重要度の高い順。

  • 不要コード・デッドコード・コメント・余分な改行は必ず削除（彼の最頻出指摘）。

    「これは使ってないかと思います。」「不使用なら削除」「余分な改行なくす」 「またこのファイル入っちゃってます。」 — 認証鍵ファイル（serviceAccount.json）の誤コミット指摘

  • 層構造（usecase / repository / query / handler）を厳守させる。usecaseは薄く、modelは厚く。コントローラー層にロジックを載せない。

    「useCaseは基本的に何もせず、model層のメソッドを呼び出すだけ」 「handlerのファイルを分割している単位がわからないので、一つにまとめるか、パスごとに切った方が良さそうです。」 「やるべきことのメソッドだけポンポンポン並んでるのが綺麗で読みやすい。ごちゃごちゃしてきたり極度に if が多いなと思ったら、本当にここに書くべき内容か考えた方がいい」

  • 1メソッド1責務／影響を閉じ込める。ある概念の余波を周辺メソッドが各々気にする実装を嫌う。

    「1つのメソッドは基本1つのことしかやっちゃダメ。例えば"消防文言"という概念の余波を、他のメソッドABCが全部気にしながら実装するのはむずい。影響はそれを管理する関数・コンポーネントに閉じ込める」

  • 取得・整形はバックエンドに寄せる。フロントでの結合・整形は重くしUXを損なう。

    「フロントで結合・整形するとフロントが重くなりUXが損なわれる。システム的な鉄則はバックエンドが頑張るスタイル」

  • 参照はコネクション（外部キー制約）で貼らせる。IDだけ持って自前突合する設計を嫌う。

    「IDだけ持って自分で突合するロジックを書くとクエリ数が増え、これ合ってるのかという怪しさも出る」 「（削除時の制約は）悪い挙動ではなく、不整合を防ぐ良い挙動。RDBの外部キー制約と同じ。使わないとAppSync/ORMの旨味が消える」

  • 呼び出し側に内部ロジックを意識させない。引数・前提が暗黙だと使う難易度が上がる。

    「この変数の中身が何かを呼び出す側が把握してないと使えないメソッドは、使う難易度が上がる。特にJSは必要な変数が足りないのに呼んだ、というバグを生みやすい」

  • DRY・集約を好む。既存ライブラリ・共通関数を優先し、同じことを何度も実装させない。

    「個別にメソッド作るのではなく、〔別案件のコード〕みたいにGetならGetでメソッドまとめた方が好みです。」 「書いた場所は全部テストで確かめる必要があるので、書けば書くほど煩わしさを伴う。共通関数を使った方がいい。同じことは何回も実装しない」

  • 命名規約を言語ごとに使い分けさせる。

    「goの引数はキャメルケースで書くことが多いです。」「indicatorCategoryId → indicatorCategoryID」（Goの頭字語は大文字） 「スネークケースが正しいです。」 — APIフィールド（snake_case）とGo（camelCase）を区別

  • コードは意味を理解して使わせる。検索で出たコードを意味も分からず貼るのを戒める。

    「出てきて使うのは悪くないけど、意味分かってないのに使うのはやめた方がいい」

  • 実データの使われ方からエッジケースを推論する。

    「後から入力忘れで、順序が逆転することがあるので、前後チェックより、重複してないチェックのほうが良い」 「これそのカレンダーの期間中のプロジェクトだけ取得できてますか？なんか全部取得していそうに見えました。」

  • 設定・秘匿情報・デプロイ安全性。ハードコードと秘匿情報コミットを許さない。

    「これはこのままだとデプロイできないので、環境変数から取得した方が良さそうです。」 「ローカルなら*で良いですが、デプロイ環境はちゃんと絞りたいですね。」（CORS） 秘匿情報がコミットされた際は git reset + force push での履歴抹消まで指示する

  • データの持ち方の改善も提案する（ただしテスト・修正コストを見て現状維持を選ぶ判断もする）。

    「ちょっとそこまで直すとテストが大変なので一旦いっか」

  • テストの抜けを指摘する（粒度は別ファイルのテスト仕様レビューを参照）。「Getのテストがないです。」

フロントエンド（Vue/Nuxt）でのレビュー観点

補助的なパターン（クライアントC の Nuxt/Vue + AWS Amplify GraphQL 案件で観測）。Go 側の観点（層分離・命名・不要コード・エラー層・横串）がフロントでもそのまま現れ、加えて以下が特徴。

  • 永続化・データ処理は composable（repository層）に切り出させる。ページ/コンポーネントに DB 処理を直書きさせない。UI内のロジックは表示の出し分けのみ。

    「基本的に永続化処理（repository層）は別ファイルに切り出しましょう。」 「UIから直接GraphQL/APIを呼ばず、リポジトリ層相当はコンポーザブルに分ける。バックエンドで言うとコントローラーに全部書いてるのと同じになっちゃう」

  • ロジックは「概念的に正しい場所」に置かせる（フロント版の「横串」）。
  • 整形はデータ取得元で行わせ、後付けの再整形を避ける。複数オブジェクトのこね回しは「難読化」と呼んで戒める（がコスト次第で条件付き許容）。

    「二つのオブジェクトから整形するというのは処理が複雑になりがちなので、難読化します。…元々のデータ取得時に詰めて返してあげればスッキリします。…が、今回はこれで良いです。」

  • 命名の正確さ（フロントでも頻出）。語の意味と実装を一致させる。

    「同期ではないので、syncという単語は不適切かと思います。普通にcreateとかでよさそうです」 「saveってあまり使いませんね。…普通にaddで良いかと思います」 「アッパースネークケースは定数の書き方なので、あまり慣習的ではないですね」

  • エラーは正しい層で扱わせる。「このファイル内でerrorをもみ消すのは微妙で、呼び出し元で処理したほうが良さそうです。」
  • 勝手な値の変更（暗黙のミューテーション）に違和感を表明する。「値のトリミングって何のためにやってます？」
  • 冗長な分岐・重複構造を削らせる／チェーン・ワンライナーを重ねさせない。「map に map を重ねると分からなくなる。ワンライナーで返そうとせず変数に代入して分割」。
  • コメントは現行コードと一致させる。古いコメントは書き換え/削除させる。
  • スキーマ・権限（Amplify GraphQL の @auth）を実利用者の動きと突き合わせて検証。「店舗の人がアップロードするので、ReadOnlyAccessにもcreate等の権限が必要そうです。」
  • CSS・アクセシビリティ・パフォーマンスの細かい指摘は彼からはほぼ出ない（スタイル nits は別レビュアー）。関心は一貫してアーキテクチャ・命名・データ層の正しさ・エラーハンドリング。

型・状態管理（フロント）の規律

  • any 型を禁止する。「正しい型定義があれば any なしで取れるはず」。
  • 型定義外プロパティの動的追加を禁ずる（型崩れ）。「中身に何が入ってるか分からなくなる」。
  • 二重管理（ダブルメンテ）を排除する。他変数から導出できる値は独立 state にせず computed/算出に。「変更タイミングが同じなら変数は1つに統合。2変数ある以上それ以上持ちたくない」。
  • 変数の所有権を守らせる。外部からの直接書き換えはNG、セッター経由にさせる。
  • Vue の watch 多用を避けさせる。「props はライフサイクルが走るのに watch すると二重監視で画面が重い。setup で初回呼び＋onUpdated で再実行」。

レビューのトーンと言い回し

  • 柔らかく、しかし断定的。ブロッカーは素の命令形（「修正お願いします」「validationを行ってください。」）、それ以外は「〜が良さそうです」「〜の方が良い」「〜かと思います」で包む。
  • GitHubの suggestion ブロックで正解コードをそのまま渡すことが多い（指摘＝丸投げにしない）。
  • 断定せず問いで気づかせる。「これ定義する意味ありますか？」「消した意図はありますか？」。
  • 既存の負債は自分の責任として引き受ける。「（元のソースが悪いですが）」「（これも元からで申し訳ないですが）」。
  • 指摘には「なぜ」を箇条書きで添えて教える。
  • 「必須の指摘」と「個人の好み」を明示的に区別する。好みはブロックせずコメントのみ。

    「本当は…が好きですが、個人的な好みもあるので、コメントのみにしておきます。」 「ちょっと長いですが、愚直にこうしましょう。」 — 命名は短さより分かりやすさを取る

  • 性能が拮抗するなら可読性を取る。「読みにくい＝バグを生みやすい」。
  • 自分の不確かさを率直に開示してから方向性を示す。「ちょっと嘘言ってたらごめんなさい」（→ [[communication-style]]）。
  • 絵文字はほぼ使わない。Approve は素っ気なく「LGTM👍」が定型。条件付きApproveを多用。

    「Approveしましたが、修正あるので、それだけ対応してからマージしてください。」

  • 英語の nits: / must: のような重大度プレフィクスは使わず、重大度は言い回しで伝える。

Git・PR運用の標準

  • mainブランチを2回取り込む：開発前（ブランチを切る前）と、PRを出す前（プッシュ前）。後からのコンフリクト手戻りを避け、CI のテストを最新main相手に走らせるため。
  • feature → feature のマージを避ける（squashでコミットが潰れコンフリクトの原因に）。
  • PRは小さく（レビューコスト最優先）。「PR大きすぎて、レビューコストが高すぎるので、いったんマージして、気づいた時にリファクタしましょう。」
  • 作業は一つずつ終わらせる。
  • ブランチのベースを必ず確認（頻出の取りこぼし）。「ブランチがdevelopに向いてました。mainが正しいです。」
  • CIでビルドを必ず通す。

テストの考え方（実行・運用面）

  • 業務フローを端から端まで通す。1ステップだけのテストを認めない。案件βでは「見積もり→受注→発注→入荷→出荷→検収→月締めがひとまとまり」と一連を要求。
  • シナリオテストはバグゼロが前提（顧客の最終確認だから）。バグチェック／モンキーテストとは区別する。
  • ハッピーパスだけでなくネガティブチェックを要求。
  • テストコードの依存・安全性も見る。前のテストに依存する構成、本番DBを操作しうるテストサーバ、SQL直書きを保守性・安全性リスクとして指摘。「万が一本番で使われたらDB何でも操作できちゃうので怖い」。
  • 詳細なテスト仕様書のレビュー観点は [[requirements-and-design]] を参照。

本番・運用・インフラの規律

  • 本番は気軽に触らせない。マージ・操作の権限を明確にゲートする。「本番に対する意識あらためてください。気軽に操作して良いものではないです。」
  • dev=本番の状態にしてから確認させる。「develop=prdの状態にしておかないと…チェックできていないバグを踏むリスクが高まります」。
  • デプロイ済みのものへの破壊的変更を禁ずる。
  • 障害報告は事象でなく原因を書かせる。タイムスタンプ付きの証跡と、原因／補足／調査内容／一時対応／恒久対応の構造で詰める。

    「話しているのは事象であって、原因が書かれてないため、変更内容が妥当かの判断がこの情報だけだとつかない」

  • マネージド基盤の運用ルールを恒久化する（例：「作成したリソースを管理コンソールから更新してはいけない」）。

接する際の注意点（コードレビュー・開発編）

  • PRを出す前に自分で1行ずつ読む。機械的ミスはエディタ設定／フォーマッタで根絶する。
  • 一度受けた指摘は記録して二度と出さない。
  • 指摘には必ず返信する（「done」「修正しました」だけでも可）。
  • 指摘は横串で直す。1箇所直して終わりにしない。
  • ロジック・整形はデータ取得元／正しい層に置く。後付けの再整形（=「難読化」）を避ける。フロントなら永続化処理は composable に切り出す。
  • 命名は意味と一致させる（sync/save/add の誤用、ケースの不統一に注意）。
  • mainを2回取り込む（ブランチ前・PR前）、PRは小さく、ブランチのベースを確認。
  • 本番には気軽に触らない。dev=本番にしてからテストする。
  • 障害は「事象」でなく「原因」まで突き止めて報告する。

データ不足（推測しない領域）

  • コードレビューは Go が最も深く、次いで Nuxt/Vue フロントでも活発。Elixir には本人レビューがほぼ無く未収集。
  • CSS・アクセシビリティ・パフォーマンスの細かい指摘は本人からはほぼ出ない。アーキテクチャ・命名・データ層・エラーハンドリング・型の健全さ（any禁止・型崩れ防止）・状態管理に集中する。
  • セキュリティ観点の自前の指摘は限定的（CORS絞り込み・秘匿情報の非コミット・環境変数化・Amplify の権限確認が中心）。体系的なセキュリティレビューは自動化AIチェックリスト側に寄っている。
  • パフォーマンス/N+1の直接指摘は少ない。性能懸念はタイムアウト・ページネーション・ORDER BY安定性といった間接的な形で現れる。
  • ユニットテスト／テストピラミッド／TDDの哲学はあまり語られない。シナリオ/結合/手動テストの比重が高い。
  • 命名規約・可読性・技術的負債を抽象概念として論じる発言はほぼなく、具体的なPR指摘として現れる。
